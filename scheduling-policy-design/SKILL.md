# 调度策略的"工作负载假设-指标-策略"推导框架

## R — 原文 (Reading)

> "How should we develop a basic framework for thinking about scheduling policies? What are the key assumptions? What metrics are important? What basic approaches have been used in the earliest of computer systems?"
>
> — Arpaci-Dusseau, OSTEP Chapter 7

---

## I — 方法论骨架 (Interpretation)

这个框架提供了一条从"工作负载长什么样"到"该选什么调度策略"的系统化推导路径，而非罗列算法清单。

核心思想是：任何调度策略都是一种在特定假设下的优化解。如果假设变了，最优策略几乎必然变。

推导链如下：
1. 先刻画工作负载的四个维度——作业到达模式（批量到达还是交互式突发）、运行时间分布（已知/未知、长/短尾）、I/O 密集度、是否可抢占。
2. 再选定一个主导优化指标——周转时间（从提交到完成的总时长）、响应时间（从提交到首次运行的时间）、公平性（各任务获得 CPU 的比例）、吞吐量（单位时间完成作业数）、或截止时间满足率（实时系统）。
3. 根据指标选择基础策略：最小化平均周转时间且运行时间已知 → SJF（最短作业优先）或其抢占版 SRTF；最小化响应时间且需要公平分享 → RR（时间片轮转）；运行时间未知且需要兼顾周转与响应 → MLFQ（多级反馈队列）；追求比例公平且任务数动态变化 → Lottery / Stride / CFS（完全公平调度器）。
4. 逐步放宽不切实际的假设。例如，SJF 假设运行时间先知，现实中用 MLFQ 通过历史执行行为自适应地近似 SJF 的效果。
5. 进入多处理器场景后，引入缓存亲和性（Cache Affinity）和负载均衡（Load Balance）作为额外硬约束。此时单队列（SQMS）简单但扩展性差，多队列（MQMS）扩展性好但负载不均衡风险高，需要迁移（migration）或工作窃取（work stealing）来补救。

## A1 — 书中的应用 (Past Application)

### 案例 1: xv6 的 Round-Robin 调度器
- **问题**: 如何在教学级操作系统中展示 CPU 虚拟化的核心机制？
- **方法论的使用**: 作者假设工作负载是一组交互式进程，优化指标是响应时间和公平性。选择 RR 作为基础策略，配合定时器中断实现抢占式上下文切换。
- **结论**: RR 以固定时间片轮转，保证每个进程都能定期获得 CPU，响应时间有界。
- **结果**: xv6 的 RR 实现成为理解"受限直接执行 + 中断驱动调度"的最小可运行模型，但也暴露了 RR 在周转时间上的劣势——长进程被频繁打断，完成时间变长。

### 案例 2: Linux CFS 的红黑树设计
- **问题**: 如何在数万任务的生产环境中实现比例公平调度，同时保持 O(log n) 的可扩展性？
- **方法论的使用**: 作者将公平性定义为"虚拟运行时间（vruntime）的最小化差异"。每个任务按实际运行时间加权计算 vruntime，调度器总是选择 vruntime 最小的任务执行。
- **结论**: 公平性指标被精确量化为 vruntime 的单调增长，调度策略转化为数据结构维护问题。
- **结果**: CFS 用红黑树维护可运行任务，避免了传统时间片带来的边界抖动问题，成为 Linux 的默认调度器，支持从桌面到服务器的广泛工作负载。

### 案例 3: MLFQ 对 SJF 假设的放宽
- **问题**: 运行时间未知时，如何同时优化响应时间和周转时间？
- **方法论的使用**: OSTEP 作者设计了一个多级反馈队列（MLFQ），用历史行为推断作业类型。短作业（高 I/O）停留在高优先级队列获得快速响应；长作业逐步降级到低优先级队列，减少对其他作业的干扰。
- **结论**: 当"运行时间已知"这一假设不成立时，用反馈机制自适应地逼近 SJF 的效果。
- **结果**: MLFQ 成为许多交互式系统（包括早期 Unix）的调度基础，它不需要先验知识，就能在响应时间和周转时间之间取得实用平衡。

## A2 — 触发场景 (Future Trigger) ★

### 用户会在什么情境下需要这个 skill?

1. **为批处理系统选择调度策略时**
   - 例如：用户有一个机器学习训练任务队列，作业长度差异巨大，目标是降低平均等待时间。此时需要分析运行时间分布，判断是否可用 SJF 或需要其抢占版 SRTF。

2. **为交互式服务优化响应延迟时**
   - 例如：用户在设计一个 Web 服务器或游戏服务器，需要保证用户请求的首次响应足够快，即使这会稍微拖慢整体吞吐量。此时 RR 或 MLFQ 可能比 FCFS 更合适。

3. **在多核/多处理器系统上设计调度器时**
   - 例如：用户正在实现一个线程池或分布式执行引擎，需要决定用单队列还是多队列，以及如何处理负载不均衡和缓存亲和性冲突。

4. **在实时系统中满足截止时间约束时**
   - 例如：用户有嵌入式控制任务，每个任务有明确的截止时间（deadline），需要选择 EDF（最早截止时间优先）或 Rate-Monotonic 调度，并分析可调度性边界。

5. **比较不同调度算法的优劣并做量化论证时**
   - 例如：用户在技术评审中需要解释"为什么我们用 CFS 而不是简单的时间片轮转"，要求给出基于指标和工作负载假设的推导。

### 语言信号 (用户的话里出现这些就应激活)

- "周转时间和响应时间怎么权衡"
- "该用 SJF 还是 RR"
- "时间片设多大合适"
- "MLFQ 的优先级反转问题"
- "多核调度怎么保证缓存亲和性"
- "CFS 的 vruntime 是怎么算的"
- "批处理队列的调度策略选型"
- "实时任务的 deadline 调度"

### 与相邻 skill 的区分

- 与 `定量性能分析框架` 的区别: 本 skill 专注于调度领域，提供从工作负载假设到策略选择的推导链；定量性能分析框架是更通用的测量与建模方法论，不限于调度。
- 与 `并发与容错设计原则` 的区别: 并发 skill 关注正确性（竞态条件、死锁、一致性），本 skill 关注性能指标（周转时间、响应时间、公平性）在调度策略中的优化。
- 与 `局部性驱动的优化决策` 的区别: 局部性 skill 关注存储层次和数据布局；本 skill 在调度语境下借用"缓存亲和性"概念，但核心是 CPU 时间分配策略。

## E — 可执行步骤 (Execution)

当 skill 被激活后，agent 应按以下步骤执行:

1. **明确工作负载假设**
   - 与用户一起确认四个维度：到达模式（批量/交互/实时突发）、运行时间分布（已知/未知、长尾/均匀）、I/O 密集度（CPU-bound / I/O-bound 比例）、可抢占性（是否允许中断）。
   - 完成标准: 能清晰描述"任务是什么样的"，并指出哪些假设是确定的、哪些是不确定的。
   - 判停条件: 若用户无法描述工作负载，则先给出典型场景的分类（如 Web 请求、ML 训练、实时控制），让用户对号入座，再进入下一步。

2. **锁定主导优化指标**
   - 在周转时间（Turnaround Time = 完成时间 - 到达时间）、响应时间（Response Time = 首次运行 - 到达时间）、公平性（Fairness = 实际 CPU 份额 / 理想份额）、吞吐量（Throughput = 完成作业数 / 单位时间）、截止时间满足率（Deadline Meet Ratio）中，帮助用户选择一个或两个主导指标。
   - 完成标准: 用户明确说出"我最关心 X"，并理解不同指标之间的冲突（如最小化周转时间通常以牺牲响应时间为代价）。
   - 判停条件: 若用户要求同时优化所有指标，则明确告知冲突关系，并建议分场景或分队列处理（如 MLFQ 的分层思想）。

3. **推导基础策略**
   - 根据前两步的结论，给出策略推荐和推导理由：
     - 周转时间优先 + 运行时间已知 → SJF / SRTF
     - 响应时间优先 + 交互式负载 → RR（需讨论时间片大小与上下文切换开销的权衡）
     - 运行时间未知 + 兼顾周转与响应 → MLFQ
     - 比例公平 + 动态任务数 → Lottery / Stride / CFS
     - 实时系统 + 硬截止时间 → EDF / Rate-Monotonic
   - 完成标准: 用户理解推荐策略与指标/假设之间的因果关系。

4. **讨论策略的边界条件与实现细节**
   - 针对推荐策略，深入一个关键实现细节：
     - RR: 时间片长度如何设定（通常为目标响应时间的 1/n，且远大于上下文切换开销）
     - MLFQ: 如何避免饥饿（aging 机制）、如何设定时间片倍数（如高优先级 1 个单位，次高 2 个，再下 4 个...）
     - CFS: vruntime 的计算公式（vruntime += delta_exec * NICE_0_LOAD / se->load.weight）、红黑树的插入/删除复杂度
     - 多处理器: 单队列（SQMS）vs 多队列（MQMS）的权衡，缓存亲和性的保持策略（如尽量将任务绑定到上次运行的核心）
   - 完成标准: 用户获得至少一个可落地的实现参数或设计决策。

5. **给出边界警告与反例**
   - 引用 counter-examples 和书中警告，提醒用户以下失败模式：
     - 更多线程/队列总是更好 → 上下文切换和缓存失效会主导性能（ce14）
     - 时间片过小 → 上下文切换开销占比激增，吞吐量暴跌
     - 忽视缓存亲和性 → 多核调度中频繁迁移任务导致 L1/L2/L3 命中率骤降
     - 用 RR 处理纯批处理负载 → 平均周转时间显著劣于 SJF
   - 完成标准: 用户意识到至少一个潜在陷阱，并知道如何监控或避免。

## B — 边界 (Boundary) ★

### 不要在以下情况使用此 skill

- **纯概念查询**: 用户只是问"什么是 CFS"或"MLFQ 有几层"，没有实际的选型或设计需求。
- **非调度场景的资源分配**: 例如内存分配、磁盘分区、网络带宽分配，这些虽然也有"分配"概念，但不涉及 CPU 时间片的抢占、上下文切换和周转/响应时间权衡。
- **个人时间管理/日程规划**: 虽然借用"调度"一词，但完全不涉及操作系统层面的技术决策。

### 作者在书中警告的失败模式

- **认为更多线程总是更好** (ce14): OSTEP 明确指出，当线程数超过物理核心数时，上下文切换开销和缓存失效（cache thrashing）会主导性能，导致"并发反扩展"（negative scaling）。这对多队列调度器设计尤为重要——队列数不等于核心数时可能适得其反。
- **时间片过小导致切换开销主导**: 在 RR 中，若时间片长度接近或小于上下文切换时间，系统会花费大量时间在保存/恢复寄存器状态上，实际吞吐量趋近于零。这是调度策略设计中必须量化的基本权衡。
- **忽视缓存亲和性导致性能断崖**: 多处理器调度中，频繁迁移任务会使该任务失去已预热的高速缓存状态，后续执行时间显著增加。OSTEP 强调：调度器应尽可能让任务在"同一个 CPU 上运行"（cache affinity）。

### 作者的盲点 / 时代局限

- 教材中的调度案例主要围绕单节点 CPU 调度，对现代 NUMA 架构、GPU 调度、Kubernetes 容器调度、无服务器（Serverless）冷启动调度等新兴场景覆盖不足。
- 对能耗感知调度（energy-aware scheduling）和异构多核（big.LITTLE）的讨论较浅，而这些已是移动设备和数据中心的关键设计维度。
- 作者假设调度开销"远小于"任务执行时间，但在微秒级延迟要求的场景（如高频交易、DPDK 网络包处理）中，调度器本身可能成为瓶颈。

### 容易混淆的邻近方法论

- **定量性能分析框架**: 该框架提供通用的性能测量和建模方法（如 Amdahl 定律、CPU Time = IC × CPI × T），可用于评估调度器效果，但不提供从工作负载到策略的推导路径。
- **并发与容错设计原则**: 关注正确性（互斥、死锁、一致性），而本 skill 关注性能指标的优化。两者在"多线程调度"场景可能同时出现，但焦点不同。
